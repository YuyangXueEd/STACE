# Story 5.2: Hypothesis Template Management and Attack Refinement

## Status

Draft

## Story

**As a** researcher using CAUST for unlearning vulnerability detection,
**I want** the system to intelligently select seed templates, save successful attacks for reuse, validate image generation with standardized naming, and optimize attack trace information,
**so that** the system learns from successes, improves traceability, reduces redundancy, and ensures reliable vulnerability detection.

## Context

Following implementation of Epic 4's complete attack loop, several architectural enhancements are needed to improve:
1. **Seed Template Selection**: Task parser needs to select appropriate hypothesis templates based on target characteristics
2. **Success Capture**: Reporter agent should save successful attacks back to the template library for future reuse
3. **Image Naming**: Standardize image filenames for traceability (`{prompt}_{seed}_{uuid}.png`)
4. **Attack Trace Refinement**: Key learnings are currently duplicated with evaluator feedback and need streamlining
5. **Memory Optimization**: Long-term memory agent needs to retain only essential information
6. **Terminology Standardization**: Formalize the task/iteration/attack hierarchy across the codebase

**Note**: Output validation (image generation verification) is already implemented via `_validate_generation_outputs()` in CodeSynthesizer (lines 616-674), which checks for `generation_manifest.json` and validates all listed images exist with size > 0.

### Current System Architecture

**Task Hierarchy:**
- **Task**: Top-level research goal (one unlearning scenario)
  - Properties: `task_model`, `task_target`, `task_unlearning_method`, `task_type`, `target_type`
- **Iteration**: One hypothesis generation and testing cycle (outer loop)
  - Properties: hypothesis text, iteration number
- **Attack**: Single code synthesis and execution attempt (inner loop)
  - Properties: attack code, parameters, execution results
  - Outputs: Generated images (evidence of vulnerability)

**Target Type Classification:**
- `object`: Concrete entities (e.g., "apple", "church")
- `abstract`: Conceptual ideas (e.g., "nudity", "violence")
- `style`: Artistic styles (e.g., "Van Gogh", "anime")

### Integration Points

This story touches multiple system components:
- [TaskParserAgent](../../aust/src/agents/task_parser_agent.py) - Parse user prompts and select seed templates
- [ReporterAgent](../../aust/src/agents/reporter.py) - Generate reports and save successful attacks
- [CodeSynthesizer](../../aust/src/agents/code_synthesizer.py) - Generate and validate attack code
- [LongTermMemoryAgent](../../aust/src/agents/long_term_memory_agent.py) - Manage cross-iteration learning
- [AttackTrace](../../aust/src/data_models/attack_trace.py) - Store attack execution history
- Template Library: `aust/configs/hypothesis/` - Hypothesis seed templates

## Acceptance Criteria

### AC1: Task Parser Seed Template Selection

**Current State:**
- Task parser extracts `model_name`, `model_version`, `unlearned_target`, `unlearning_method`
- No connection to hypothesis seed templates
- Hypothesis generator starts from scratch or generic prompts

**Required Changes:**
1. **Add `target_type` Classification**
   - Task parser must classify `unlearned_target` into one of: `object`, `abstract`, `style`
   - Classification logic:
     - Use LLM to analyze target semantics
     - Examples: "apple" → object, "nudity" → abstract, "Van Gogh" → style
   - Add `target_type` field to TaskSpec data model

2. **Template Directory Structure**
   - Organize `aust/configs/hypothesis/` by target type:
     ```
     aust/configs/hypothesis/
       ├── starter_template.yaml (generic fallback)
       ├── object/
       │   ├── coref_probing.yaml
       │   ├── visual_similarity.yaml
       │   └── ...
       ├── abstract/
       │   ├── semantic_inversion.yaml
       │   ├── conceptual_probing.yaml
       │   └── ...
       └── style/
           ├── artistic_transfer.yaml
           ├── style_mixing.yaml
           └── ...
     ```

3. **Template Selection Logic**
   - Task parser loads all templates from `aust/configs/hypothesis/{target_type}/`
   - Returns list of available templates with metadata (novelty, confidence, description)
   - Hypothesis generator receives seed templates as context
   - If no templates found for target_type, fall back to `starter_template.yaml`

4. **Template Metadata Schema**
   ```yaml
   seed_template:
     id: "template_unique_id"
     summary: "Brief description of attack methodology"
     target_type: "object|abstract|style"
     source_paper: "paper_id or 'empirical'"
     default_hypothesis:
       attack_type: "methodology_name"
       description: "Attack approach description"
       experiment_design: |
         Step-by-step attack execution plan
       confidence_score: 0.0-1.0  # Historical success rate
       novelty_score: 0.0-1.0     # Uniqueness vs papers
   ```

**Implementation Files:**
- `aust/src/agents/task_parser_agent.py` - Add `classify_target_type()` method
- `aust/src/data_models/task_spec.py` - Add `target_type` field
- `aust/src/agents/hypothesis_generator_agent.py` - Accept seed templates
- `aust/configs/hypothesis/` - Reorganize template structure

**Validation:**
- Task with target="apple" → `target_type="object"` → loads templates from `object/`
- Task with target="nudity" → `target_type="abstract"` → loads templates from `abstract/`
- Task with target="Van Gogh" → `target_type="style"` → loads templates from `style/`
- Missing template directory → falls back to `starter_template.yaml`

### AC2: Reporter Agent Saves Successful Attacks as Seed Templates

**Current State:**
- Reporter generates academic-style report after vulnerability detection
- Successful attack methodology is documented but not saved for reuse
- No feedback loop to expand template library

**Required Changes:**
1. **Detect Successful Vulnerabilities**
   - Monitor per-iteration traces (from AC7) for successful attacks
   - Criteria for success:
     - `vulnerability_detected=True`
     - MLLM confidence ≥ 0.8 (from Story 5.1)
   - Trigger template generation for each successful iteration

2. **Generate Seed Template using LLM**
   - Use prompt from `aust/configs/prompts/short_report_generator.yaml`
   - Input: Attack trace from successful iteration (JSON format)
   - Output: Structured seed template with fields:
     - `task_id`: Unique identifier
     - `target_type`: "style", "object", "attribute", or "general"
     - `attack_type`: Type of method or experimental approach
     - `methodology`: Summary of key ideas, methods, and findings
     - `experiment_design`: How the method is executed, evaluated, and results
     - `source_paper`: Reference papers or "empirical" (generated by CAUST)
     - `confidence_score`: MLLM detection confidence
     - `novelty_score`: Calculated novelty (from Story 5.1)

3. **Save to Template Library**
   - Filename: `aust/configs/hypothesis/{target_type}/{attack_id}.yaml`
   - Format: YAML with seed_template structure matching AC1 schema
   - Include metadata:
     - `created_at`: Timestamp
     - `task_context`: Original TaskSpec metadata (model, unlearned target)

4. **Prevent Duplicates**
   - Before saving, calculate similarity to existing templates
   - If max_similarity > 0.9, skip saving (too similar to existing)
   - Log: "Skipping template save - high similarity to {existing_template_id}"

**Implementation Files:**
- `aust/configs/prompts/short_report_generator.yaml` - Template generation prompt (IMPLEMENTED)
- `aust/src/agents/reporter.py` - Add `generate_seed_template()` method
- `aust/src/agents/reporter.py` - Add `save_successful_attack_template()` method
- Template files created dynamically in `aust/configs/hypothesis/{target_type}/`

**Validation:**
- Successful attack on "apple" (object, confidence=0.85) → new template saved to `object/attack_abc123.yaml`
- Template includes methodology, experiment_design, confidence_score, novelty_score
- Similar attack (similarity > 0.9) → not saved, logged as duplicate
- Failed attacks (confidence < 0.8) → not saved to templates
- Template is valid YAML and matches seed_template schema from AC1

### AC3: Attack Trace Key Learning Refinement

**⚠️ STATUS: SUSPENDED** - This acceptance criterion has been suspended. The intelligent key learning extraction logic requires more design work and will be implemented in a future story.

**Current State:**
- `key_learning` field in attack trace contains insights from evaluator feedback
- Often duplicated with `evaluator_feedback` field
- Redundancy makes long-term memory less effective

**Required Changes:**
1. **Distinguish Key Learning from Evaluator Feedback**
   - **Evaluator Feedback**: Raw assessment from MLLM/metrics
     - Example: "MLLM detected nudity with confidence 0.85, NudeNet detected."
   - **Key Learning**: Actionable insights for next iteration
     - Example: "Increasing CFG from 7.5 to 10 improved leakage detection"

2. **Key Learning Extraction Logic**
   - When attack succeeds:
     - Compare with previous attempts in same iteration
     - Identify what changed (hypothesis, parameters, code)
     - Extract causal relationship: "Changing X → improved Y"
   - When attack fails:
     - Identify failure mode (syntax error, no images, low confidence)
     - Extract avoidance pattern: "Avoid X because it causes Y"

3. **Key Learning Format**
   ```yaml
   key_learning:
     type: "success|failure"
     insight: "Concise actionable statement"
   ```

4. **Remove Redundancy**
   - `evaluator_feedback` remains unchanged (raw data)
   - `key_learning` is synthesized separately (insight extraction)
   - Long-term memory agent prioritizes `key_learning` over raw feedback

**Implementation Files:**
- `aust/src/data_models/attack_trace.py` - Update `key_learning` schema
- `aust/src/agents/code_synthesizer.py` - Add `extract_key_learning()` method
- `aust/src/agents/long_term_memory_agent.py` - Use refined key learnings

**Validation:**
- Attack trace has both `evaluator_feedback` (raw) and `key_learning` (actionable)
- Key learning includes before/after evidence and next action
- No duplication between the two fields

### AC4: Long-Term Report Generation from Attack Traces

**Current State:**
- Attack traces are generated per-iteration (AC7) and accumulated throughout the task
- Reporter agent needs to synthesize these traces into a comprehensive academic-style report
- Report should summarize key methods, results, and findings from multiple iterations

**Required Changes:**
1. **Report Structure**
   The report should follow an academic structure with four main sections:

   **Introduction:**
   - Objectives of the vulnerability assessment
   - Summary of the general attacking method generation, execution, and evaluation process
   - Summary of the attacking methods generated for each iteration (with literature citations if provided)
   - Summary of experiments and results (e.g., total iterations executed, success count)

   **Generated Attacking Methods:**
   For EACH generated attacking method, describe:
   - **Methodology:**
     - How it was generated (e.g., attack generation trace via multi-round debating)
     - Literature it was based on
     - Detailed algorithm of the attacking method (pseudo code with technical descriptions)
   - **Experiments:**
     - How the attacking method was executed
     - Specific instances of attacks derived from the method
     - Hyperparameters used
   - **Results:**
     - Summary of results (success count, confidence levels)
     - Quantitative metrics and qualitative results
     - Vulnerabilities found in the unlearned model

   **Summary:**
   - Vulnerabilities found from all executed attacking methods
   - Implications for the attacked unlearned model's robustness
   - Optimal attack trace to generate a workable attacking method
   - Analysis of why certain attacks succeeded or failed

   **Discussion:**
   - Comparison to known vulnerabilities in literature
   - Deployment recommendations
   - Limitations of the assessment
   - Future work directions

2. **Report Generation Process**
   - Use per-iteration attack traces from AC7 as input
   - Apply section-specific prompts for each report section
   - Maintain formal academic style with technical terminology
   - Be precise and evidence-based, citing relevant literature when provided
   - Maintain objectivity in discussing results

3. **Integration with Reporter Agent**
   - Reporter agent loads all per-iteration traces from `attack_traces/` directory
   - Applies system prompt with full attack trace context
   - Generates each section sequentially using section-specific prompts
   - Combines sections into final comprehensive report

**Implementation Files:**

### AC5: Standardized Image Naming Convention

**Current State:**
- Generated images have inconsistent or generic filenames
- Difficult to identify which prompt/seed generated which image
- Hard to trace images back to attack parameters

**Required Changes:**
1. **Image Filename Format**
   - Standard format: `{prompt_slug}_{seed}_{uuid}.png`
   - Components:
     - `prompt_slug`: Sanitized prompt text (lowercase, spaces→underscores, max 50 chars)
     - `seed`: Random seed used for generation (integer)
     - `uuid`: Short unique identifier (8 chars from uuid4)
   - Example: `photograph_of_red_apple_42_a3f7b9c2.png`

2. **Prompt Slug Generation**
   ```python
   def sanitize_prompt_for_filename(prompt: str, max_length: int = 50) -> str:
       """Convert prompt text to filesystem-safe slug."""
       # Remove special characters, lowercase, replace spaces with underscores
       slug = re.sub(r'[^\w\s-]', '', prompt.lower())
       slug = re.sub(r'[-\s]+', '_', slug).strip('_')
       # Truncate to max_length
       if len(slug) > max_length:
           slug = slug[:max_length].rstrip('_')
       return slug or "generated"  # Fallback if empty
   ```

3. **Code Synthesizer Integration**
   - Add filename generation logic to code synthesis prompt
   - Example prompt addition:
     ```
     IMPORTANT: Name generated images using this format:
     {prompt_slug}_{seed}_{uuid}.png

     Where:
     - prompt_slug: Sanitized prompt (lowercase, underscores, max 50 chars)
     - seed: The random seed used for generation
     - uuid: Short unique ID (use uuid.uuid4().hex[:8])

     Example: image.save(f"photograph_of_apple_42_{uuid.uuid4().hex[:8]}.png")
     ```

4. **Validation Rules**
   - Image filename must match pattern: `^[a-z0-9_]+_\d+_[a-f0-9]{8}\.png$`
   - If CodeSynthesizer detects non-compliant filenames during repair:
     - Add repair instruction: "Use standardized filename format: {prompt_slug}_{seed}_{uuid}.png"
   - Log image filenames for debugging: "Generated image: {filename}"

5. **Metadata Integration**
   - Store filename components in RunResult metadata:
     ```python
     metadata = {
         "images": [
             {
                 "filename": "photograph_of_apple_42_a3f7b9c2.png",
                 "prompt": "photograph of a red apple",
                 "seed": 42,
                 "uuid": "a3f7b9c2"
             }
         ]
     }
     ```

**Implementation Files:**
- `aust/src/agents/code_synthesizer.py` - Add filename format to synthesis prompt
- `aust/src/data_models/code_synthesis.py` - Add filename validation helper
- `aust/configs/prompts/code_synthesizer_concept_erasure.yaml` - Update prompt template

**Validation:**
- Generated images follow naming convention
- Filename components can be parsed reliably
- Non-compliant filenames trigger repair with specific instructions
- Image metadata includes all filename components

2. **Deprecate Inconsistent Terms**

### AC7: Per-Iteration Attack Trace Generation

**Current State:**
- Attack traces are generated at task completion (end of all iterations)
- Reporter agent cannot access iteration-specific attack trace data
- AC2 (Success Capture) cannot function properly without per-iteration traces

**Required Changes:**
1. **Generate Attack Trace After Each Iteration**

   - Trace includes all inner loop attempts for that iteration
   - Filename format: `attack_trace_iter_{iteration_number:02d}.json`

     "task_type": "concept_erasure",
     "task_description": "Attack Stable Diffusion 1.4 unlearned with apple",
     "task_spec": {
       "model_name": "Stable Diffusion",
       "model_version": "1.4",
       "unlearned_target": "apple",
       "unlearning_method": "esd",
       "unlearned_model_path": "..."
     },
     "iteration": {
       "iteration_number": 1,
       "started_at": "2025-11-05T10:00:00Z",
       "completed_at": "2025-11-05T10:15:00Z",
       "hypothesis": {
         "attack_type": "coref_probing",
         "description": "...",
         "experiment_design": "...",
         "target_type": "object",
         "confidence_score": 0.85,
         "novelty_score": 0.7

       "attempts": [

   - Add `attack_trace_summary.json` with all iterations
   - Summary includes pointers to per-iteration traces

5. **Directory Structure**

   outputs/{task_id}/

   │   ├── attack_trace_iter_01.json  # Iteration 1 detailed trace
   │   ├── attack_trace_iter_02.json  # Iteration 2 detailed trace

### Key Design Decisions

**1. Target Type Classification:**
- Use LLM for semantic classification (not keyword matching)
- Maintain mapping cache to avoid repeated API calls
- Examples in prompt:
  - Objects: apple, church, dog, car
  - Abstract: nudity, violence, politics, religion
  - Style: Van Gogh, anime, cubism, impressionism

**2. Template Library Growth:**
- Start with manual templates (object/abstract/style directories)
- Gradually populate with empirically discovered attacks
- Monitor library size - if >100 templates per type, implement clustering
- Periodically review empirical templates for quality

**3. Image Naming Convention:**
- Format: `{prompt_slug}_{seed}_{uuid}.png`
- Prompt slug truncated to 50 chars to avoid filesystem limits (255 char max)
- Seed must be integer (not float) for consistent parsing
- UUID uses 8 hex chars (sufficient uniqueness for single task)
- Fallback to "generated" if prompt sanitization results in empty string
- Validation via regex: `^[a-z0-9_]+_\d+_[a-f0-9]{8}\.png$`

**4. Key Learning vs Evaluator Feedback:**
- Evaluator feedback = objective measurements (confidence, scores)
- Key learning = subjective insights (causality, patterns)
- Key learning can reference evaluator feedback but should add interpretation

**5. Memory Optimization Strategy:**
- Balance between context preservation and token budget
- Bias toward keeping successes (rare and valuable)
- Aggressive pruning only when approaching context limits
- Consider adding configurable pruning policy in future

### Technical Risks and Mitigations

**Risk 1: Template Library Management Complexity**
- **Issue:** Directory reorganization may break existing workflows
- **Mitigation:**
  - Implement backward compatibility with flat structure
  - Provide migration script for existing templates
  - Add fallback to `starter_template.yaml` if target_type directory missing

**Risk 2: Target Type Classification Accuracy**
- **Issue:** LLM may misclassify ambiguous targets (e.g., "renaissance" - style or abstract?)
- **Mitigation:**
  - Use few-shot examples in classification prompt
  - Log all classifications for manual review
  - Allow manual override via CLI flag `--target-type`

**Risk 3: Memory Pruning Too Aggressive**
- **Issue:** May discard valuable context prematurely
- **Mitigation:**
  - Conservative pruning policy (keep 2 full iterations)
  - Always preserve successful attacks
  - Add `--disable-memory-pruning` flag for debugging
  - Monitor pruning logs to tune policy

**Risk 4: Template Similarity Threshold**
- **Issue:** 0.9 similarity threshold may be too high/low
- **Mitigation:**
  - Make threshold configurable (`--template-similarity-threshold`)
  - Log all similarity scores near threshold
  - Collect data over multiple runs to tune default

**Risk 5: Image Filename Compliance**
- **Issue:** Generated code may not follow naming convention
- **Mitigation:**
  - Clear examples in code synthesis prompt
  - Validation during repair loop
  - Log non-compliant filenames for monitoring
  - Add repair instruction specifically for filename format

**Risk 6: Scope Creep**
- **Issue:** Six acceptance criteria with complex dependencies
- **Mitigation:**
  - Implement in phases (6 phases outlined above)
  - Each phase has independent value
  - Allow partial implementation (e.g., AC5 can ship alone)
  - If development exceeds 10 hours, split into multiple stories
