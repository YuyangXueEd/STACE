# Story 2.2: Vector Database & Embedding System

## Status

Completed 

## Story

**As a** developer,
**I want** to set up a Qdrant vector database with local embeddings for semantic search over paper cards,
**so that** agents can retrieve relevant research based on query similarity to support hypothesis generation.

## Background & Context

### Paper Card Assets (Story 2.1.1)
- ✅ **101 paper cards** generated in `.paper_cards/` directory
- ✅ **Structured markdown format** with sections: Metadata, Quick Summary, Research Problem, Methodology, Experiments, Results, Relevance to Our Work
- ✅ **Organized by taxonomy**: `any-to-t/` and `any-to-v/` with subdirectories by attack level

### Why Vector RAG (Not GraphRAG)
**Decision**: Use standard vector-based RAG with Qdrant for MVP (Epic 2, Week 2)

**Rationale**:
1. **Timeline**: Vector RAG can be implemented in 2-3 days vs 5-7 days for GraphRAG
2. **Use Case Fit**: Paper cards already provide structured information (methodology, experiments, results). Query Generator (Story 2.3) needs semantic search for "similar attack methods" which vector RAG handles well
3. **Performance**: Meets NFR8 (<5s retrieval) with simpler architecture
4. **Leverages Story 2.1.1**: Paper card structure is optimized for section-based chunking
5. **Future-proof**: Can add GraphRAG in Phase 2 if multi-hop reasoning becomes necessary

### Why Qdrant
**Decision**: Use Qdrant as vector database

**Rationale**:
1. **Integrated Metadata**: Qdrant natively stores structured payloads (metadata) with vectors enabling filtered search without separate JSON files
2. **Local Deployment**: Qdrant supports local in-memory mode and disk persistence for development/production
3. **Performance**: Optimized for fast similarity search with HNSW index, excellent for < 5s latency (NFR8)
4. **CAMEL-AI Integration**: CAMEL-AI's RAG examples include Qdrant storage (see `external/camel/docs/cookbooks/advanced_features/agents_with_rag.ipynb`)
5. **Local Embeddings**: Works seamlessly with SentenceTransformers (no OpenRouter API calls)
6. **Rich Filtering**: Powerful filtering DSL for section type, task type, attack level queries

## Acceptance Criteria

### 1. Qdrant Vector Database Setup
- Qdrant collection created in `rag/vector_db.py` with collection name `"aust_papers"`
- Embedding function: `sentence-transformers/all-MiniLM-L6-v2` (384-dim, local)
- Storage: Local disk persistence at `rag/vector_index/` (mounted on persistent volume)
- Distance metric: Cosine similarity

### 2. Paper Card Chunking Strategy
- **Chunking by semantic sections**: Methodology, Experiments, Results, Relevance
- Each chunk includes paper title prefix for context: `"[METHODOLOGY] {title}\n{text}"`
- Payload (metadata) stored per chunk in Qdrant:
  - `arxiv_id`: str (e.g., "2503.14232")
  - `section`: str ("METHODOLOGY", "EXPERIMENTS", "RESULTS", "RELEVANCE")
  - `task_type`: str ("any-to-t", "any-to-v")
  - `attack_level`: str ("input_level", "encoder_level", "generator_level", "output_level", "others")
  - `paper_title`: str
  - `card_path`: str (relative path to `.paper_cards/`)
  - `text`: str (full chunk text for retrieval)

### 3. Index Building Script
- Script: `scripts/build_vector_index.py`
- Reads all paper cards from `.paper_cards/` recursively
- Parses markdown to extract sections (Methodology, Experiments, Results, Relevance)
- Embeds each section using SentenceTransformers
- Upserts to Qdrant collection with payload
- Progress logging with total chunks processed
- Estimated ~400-500 chunks from 101 papers (4-5 sections per paper)

### 4. PaperRAG Query Interface
- Class `PaperRAG` in `rag/vector_db.py` with methods:
  - `search(query: str, top_k: int = 5, section_filter: Optional[str] = None, task_type_filter: Optional[str] = None) -> List[SearchResult]`
  - Returns: `[{arxiv_id, section, text, similarity_score, metadata}, ...]`
- Supports filtered search using Qdrant filter DSL:
  - `section_filter`: "METHODOLOGY", "EXPERIMENTS", "RESULTS", "RELEVANCE"
  - `task_type_filter`: "any-to-t", "any-to-v"
- Fast collection loading: < 2 seconds to initialize from disk

### 5. Search Performance (NFR8)
- Single query latency: < 5 seconds (measured on 101 papers, ~400-500 chunks)
- Batch search (3 queries for Story 2.3): < 10 seconds total
- Memory footprint: < 500MB for index + embeddings

### Key Implementation Notes

**CAMEL-AI Qdrant Storage Pattern (from cookbook):**

Based on `external/camel/docs/cookbooks/advanced_features/agents_with_rag.ipynb`, CAMEL uses Qdrant via `QdrantStorage`:

```python
from camel.storages import QdrantStorage
from camel.embeddings import OpenAIEmbedding  # We'll use SentenceTransformers instead
from camel.retrievers import VectorRetriever

# Initialize Qdrant storage
storage = QdrantStorage(
    vector_dim=384,  # all-MiniLM-L6-v2 dimension
    path="rag/vector_index",  # Local persistence directory
    collection_name="aust_papers",
)

# Use VectorRetriever for indexing and search
retriever = VectorRetriever(
    embedding_model=sentence_transformers_embedding,  # Custom embedding
    storage=storage
)
```

**For AUST, we'll adapt this pattern:**
- Use `QdrantStorage` from CAMEL with local `path` (not cloud URL)
- Replace `OpenAIEmbedding` with custom `SentenceTransformersEmbedding` wrapper
- Leverage `VectorRetriever` for consistent API with CAMEL agents

**Paper Card Parsing Strategy:**

Paper cards use markdown sections with `## Heading` format:

```markdown
## Methodology

### Core Method
...

### Key Techniques
...
```

**Chunking approach:**
1. Split by `## Section Name` (Methodology, Experiments, Key Results, Relevance)
2. Combine sub-sections (###) under parent section
3. Prepend section tag: `"[METHODOLOGY] {paper_title}\n{full_section_text}"`

**Why section-level chunking (not paragraph)?**
- Methodology sections are ~500-1000 tokens (optimal for semantic search)
- Preserves logical coherence (algorithm descriptions stay together)
- Easier filtering (Query Generator can target "METHODOLOGY" for attack methods)

**Metadata Extraction:**

Paper cards start with frontmatter-style metadata:

```markdown
# Paper Title

## Metadata
- **Authors**: ...
- **Venue**: ...
- **ArXiv ID**: 2503.14232
- **Model Type**: T→I
- **Attack Level**: input_level
```

Parse this section to populate Qdrant payload `{arxiv_id, task_type, attack_level, ...}`.

**Qdrant Filtering DSL:**

Qdrant supports complex filters via `models.Filter`:

```python
from qdrant_client.models import Filter, FieldCondition, MatchValue

# Example: Search only Methodology sections for any-to-t papers
filter_conditions = Filter(
    must=[
        FieldCondition(key="section", match=MatchValue(value="METHODOLOGY")),
        FieldCondition(key="task_type", match=MatchValue(value="any-to-t")),
    ]
)

results = qdrant_client.search(
    collection_name="aust_papers",
    query_vector=embed_query(query),
    query_filter=filter_conditions,
    limit=top_k
)
```

**Performance Considerations:**

- **Embedding caching**: Cache SentenceTransformers model in memory (loaded once)
- **Batch processing**: Upsert 50 points per Qdrant batch for optimal performance
- **HNSW index**: Qdrant uses HNSW by default (fast approximate nearest neighbor)
- **Collection persistence**: Qdrant saves to disk automatically when `path` is set

**Error Handling:**

- **Parsing failures**: Log failed paper cards, continue processing others
- **Embedding errors**: Skip chunks with embedding failures, log for manual review
- **Qdrant connection**: Retry logic with exponential backoff (3 attempts)
- **Empty sections**: Skip sections with < 50 characters (too short for meaningful search)
