# Story 1.7: Inner Loop Orchestrator

## Status

Draft

## Story

**As a** researcher,
**I want** an Inner Loop Orchestrator that coordinates the research cycle,
**so that** the system autonomously iterates until a vulnerability is discovered or max iterations reached.

## Acceptance Criteria

1. Loop Orchestrator implemented in `loop/inner_loop.py`
2. Orchestrator requires a valid `TaskSpec` from Story 1.0 before starting; on invalid input, exits gracefully with actionable error
3. Orchestrator manages loop state: iteration count, experiment history, current hypothesis, feedback
4. Loop flow (per-iteration steps 2–4):
  - Hypothesis Generator → Critic two-round micro-loop with Query Generator + RAG context
  - Code Synthesis & Experiment Execution (Story 1.6a) with self-repair retries
  - Evaluator assessment and feedback
5. Exit conditions: VULNERABILITY_FOUND OR iteration count >= 10 (configurable in `configs/loop_config.yaml`)
6. Loop state persisted to `outputs/loop_state.json` after each iteration (supports restart)
7. Attack trace generation: each iteration's hypothesis, experiment parameters, results, and feedback appended to `outputs/attack_traces/trace_{run_id}.md`
