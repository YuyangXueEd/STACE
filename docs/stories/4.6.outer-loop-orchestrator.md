# Story 4.6: Outer Loop Orchestrator

## Status

✅ **COMPLETE** - Outer loop orchestration implemented in main.py with judge feedback integration

## Story

**As a** researcher,
**I want** an Outer Loop Orchestrator that implements the 5-iteration outer loop (Step 5) and subsequent report generation with judging and memory indexing (Step 6),
**so that** the system iteratively refines attack hypotheses through 5 iterations of Steps 2-4, then generates a final report evaluated by multi-perspective judges and indexes successful attacks as experience memory.

## Acceptance Criteria

1. Outer Loop Orchestrator implemented in `loop/outer_loop.py` with two distinct phases:
   - **Phase 1 (Step 5)**: Execute 5 iterations of Steps 2-4 (Hypothesis Generation & Critic → Code Synthesis & Execution → MLLM Evaluation)
   - **Phase 2 (Step 6)**: After 5 iterations complete, generate final report → multi-perspective judging → memory indexing
2. **Step 5 Implementation**: Orchestrator repeats Steps 2-4 for 5 iterations:
   - Feed evaluation results from Step 4 back to Step 2 as context for next iteration
   - Track hypothesis evolution and improvement across iterations
   - Log all intermediate results (hypotheses, code, evaluations) for final report
3. **Step 6 Implementation**: After 5 outer loop iterations complete:
   - Call Reporter Agent to generate academic paper from all 5 iterations' interaction history
   - Call all Judge Agents (3-5 personas) to evaluate report quality
   - If attack successful: index hypothesis + attack results as "experience" in RAG (CAMEL-AI long-term memory)
   - Aggregate judge evaluations and save final output package
4. Orchestrator logs all outputs and timestamps for reproducibility (each iteration and final report)
5. Outer loop iteration budget hardcoded to 5 (not configurable for MVP consistency)
6. Final output package saved to `aust/outputs/final_{run_id}/` containing: report, attack traces from all 5 iterations (JSON + MD), all judge evaluations, aggregated summary, memory indexing confirmation

### Completion Notes List

1. **Integrated Implementation**: Outer loop orchestration implemented directly in `main.py` rather than separate module
   - Simpler architecture for MVP
   - All orchestration logic in `run_outer_loop()` function
   - Easy to refactor into separate class if needed

2. **Multi-Iteration Support**:
   - CLI flag: `--outer-iterations N` (default: 1)
   - Configurable iteration count for experimentation
   - Each outer iteration generates unique task_id: `{base_id}_outer{N:02d}`
   - TaskSpec reused across iterations for efficiency

3. **Judge Feedback Loop**:
   - Judge summary captured after committee evaluation (lines 457-466)
   - Feedback injected into `TaskSpec.metadata.outer_loop.judge_feedback`
   - Hypothesis generator can access feedback in next iteration
   - Enables iterative refinement based on judge critiques

4. **Workflow Pipeline**:
   - Inner Loop → Report → Judge → Capture Feedback → Next Iteration
   - `run_inner_loop()`: Execute hypothesis generation and testing
   - `generate_report_for_state()`: Create academic report
   - `_run_judge_committee()`: Multi-perspective evaluation
   - Feedback extraction from `summary_{task_id}.md`

5. **Output Organization**:
   - Per-task directories: `outputs/{task_id}/`
   - Reports centralized: `outputs/reports/`
   - Judgments per-task: `outputs/{task_id}/judgments/`
   - Attack traces per-task: `outputs/{task_id}/attack_traces/`
   - State persistence: `outputs/{task_id}/loop_state.json`

6. **Resume Support**:
   - `--resume-state` flag bypasses outer loop iteration handling
   - Directly resumes from saved `loop_state.json`
   - Single iteration execution in resume mode
